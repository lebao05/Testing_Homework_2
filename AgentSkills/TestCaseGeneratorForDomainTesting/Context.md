# Domain Testing Analysis - FR-03: Forgot Password and Password Reset

## 1. Requirement Analysis

FR-03 defines a two-step forgot password and password reset process.

Step 1 - Get OTP:

- The user enters the email address of an already registered account.
- The system generates a random numeric OTP and sends it by email.
- In the demo environment, the OTP is displayed directly on the screen.
- The UI must show a step indicator such as "Step 1 / 2".
- The UI must provide a "Back to Login" action.

Step 2 - Reset Password:

- The user enters OTP, new password, and confirm new password.
- The new password must satisfy the same strong password rules as FR-01.
- New password and confirm new password must match.
- OTP is valid only for the email that requested it and cannot be used for another email.

User inputs:

- Email address
- OTP / reset token
- New password
- Confirm new password

Assumptions:

- Email is required in Step 1 and is also needed in Step 2 either as an explicit API field or preserved state from Step 1.
- The API specification includes `email`, `resetToken`, and `newPassword`; the system requirement also requires confirm new password, so confirm new password is treated as a required UI input.
- OTP is treated as a string, not a number, so leading zeroes are preserved.
- Password composition rules are taken from FR-01.
- This analysis identifies input domains and equivalence classes only. It does not include boundary values or test cases.

## 2. Input Domain Identification

- Input name: Email address
  - Data type: String
  - Purpose: Identifies the registered account requesting password reset.
  - Valid domain: A syntactically valid email address that belongs to an existing registered user account.
  - Invalid domain: Empty value, missing value, null value, non-string value, malformed email format, unregistered email address.
  - Constraints: Must be usable as an email address and must map to a registered account.
  - Dependencies with other inputs: OTP must be generated for and validated against this email address.
  - Assumptions: Email uniqueness is handled by registration rules; the forgot-password flow only accepts registered emails.

- Input name: OTP / reset token
  - Data type: String of numeric digits
  - Purpose: Verifies that the user has the reset token generated by the forgot-password step.
  - Valid domain: Numeric OTP generated by the system for the same email address and still valid for reset.
  - Invalid domain: Empty value, missing value, null value, non-string value, non-numeric value, malformed OTP, expired OTP, already-used OTP, OTP not generated by the system, OTP generated for a different email.
  - Constraints: Must match the system-generated OTP for the requesting email.
  - Dependencies with other inputs: Depends on email address; the OTP is only valid for the email that requested it.
  - Assumptions: OTP should become invalid after use and should have an expiry period based on security requirement SEC-07.

- Input name: New password
  - Data type: String
  - Purpose: Defines the replacement password for the user account.
  - Valid domain: String satisfying the strong password rule: enough length, at least one uppercase letter, at least one lowercase letter, at least one digit, and at least one allowed special character from `@`, `$`, `!`, `%`, `*`, `?`, `&`.
  - Invalid domain: Empty value, missing value, null value, non-string value, weak password, password missing required character categories, password using only unsupported special characters, password containing only whitespace.
  - Constraints: Must satisfy all password strength rules from FR-01.
  - Dependencies with other inputs: Must match confirm new password; reset is only allowed when the email and OTP are valid.
  - Assumptions: Password is not displayed clearly in the UI because password fields should use `type="password"`.

- Input name: Confirm new password
  - Data type: String
  - Purpose: Confirms that the user intentionally entered the desired new password.
  - Valid domain: Same exact value as new password.
  - Invalid domain: Empty value, missing value, null value, non-string value, value different from new password.
  - Constraints: Must match new password exactly.
  - Dependencies with other inputs: Fully dependent on new password.
  - Assumptions: Confirmation is required by the UI/system requirement even if not shown in the reset API example.

## 3. Equivalence Class Partitioning

- EC1 - Valid registered email
  - Input: Email address
  - Type: Valid
  - Class: Email is syntactically valid and belongs to an existing registered account.
  - Reason: The forgot-password flow is intended only for registered users.

- EC2 - Malformed email
  - Input: Email address
  - Type: Invalid
  - Class: Email value does not follow valid email format.
  - Reason: The requirement states that the user enters an email address; form requirements also expect email validation.

- EC3 - Unregistered email
  - Input: Email address
  - Type: Invalid
  - Class: Email has valid format but does not belong to any registered account.
  - Reason: FR-03 requires the email address to be already registered.

- EC4 - Missing or empty email
  - Input: Email address
  - Type: Invalid
  - Class: Email is omitted, empty, null, or only whitespace.
  - Reason: The system cannot generate or associate an OTP without an account email.

- EC5 - Email with incorrect data type
  - Input: Email address
  - Type: Invalid
  - Class: Email is provided as a non-string value.
  - Reason: Email should be processed as text and validated as an email address.

- EC6 - Valid OTP for the same email
  - Input: OTP / reset token
  - Type: Valid
  - Class: OTP is generated by the system, has the required numeric OTP form, is still valid, unused, and belongs to the same email.
  - Reason: FR-03 states that OTP is valid only for the email that requested it.

- EC7 - OTP with invalid format
  - Input: OTP / reset token
  - Type: Invalid
  - Class: OTP is missing, empty, null, non-numeric, malformed, or wrong data type.
  - Reason: The requirement defines OTP as numeric digits generated by the system.

- EC8 - OTP not generated by the system
  - Input: OTP / reset token
  - Type: Invalid
  - Class: OTP has acceptable format but does not match any generated reset token.
  - Reason: A syntactically correct OTP is not valid unless generated by the system.

- EC9 - OTP generated for a different email
  - Input: OTP / reset token
  - Type: Invalid
  - Class: OTP is real but belongs to another email address.
  - Reason: FR-03 explicitly says OTP cannot be used for another email.

- EC10 - Expired or already-used OTP
  - Input: OTP / reset token
  - Type: Invalid
  - Class: OTP was previously valid but is no longer accepted.
  - Reason: Secure reset flows require OTP invalidation; SEC-07 states OTP should have a time limit and be invalidated after use.

- EC11 - Strong new password
  - Input: New password
  - Type: Valid
  - Class: Password satisfies all required strength rules.
  - Reason: FR-03 says new password must follow FR-01 password rules.

- EC12 - Missing or empty new password
  - Input: New password
  - Type: Invalid
  - Class: New password is omitted, empty, null, or only whitespace.
  - Reason: Password reset cannot complete without a replacement password.

- EC13 - New password with incorrect data type
  - Input: New password
  - Type: Invalid
  - Class: New password is not a string.
  - Reason: Password fields are textual secrets and must be validated as strings.

- EC14 - Weak new password
  - Input: New password
  - Type: Invalid
  - Class: Password does not satisfy one or more required strength rules.
  - Reason: FR-03 inherits the strong password policy from FR-01.

- EC15 - New password with unsupported special character usage
  - Input: New password
  - Type: Invalid
  - Class: Password relies on special characters outside the allowed set and otherwise fails the required allowed-special-character rule.
  - Reason: FR-01 lists the accepted special characters.

- EC16 - Confirm password matches new password
  - Input: Confirm new password
  - Type: Valid
  - Class: Confirm new password is exactly the same as new password.
  - Reason: FR-03 requires the two password fields to match.

- EC17 - Confirm password does not match new password
  - Input: Confirm new password
  - Type: Invalid
  - Class: Confirmation value differs from new password.
  - Reason: The system must reject mismatched password confirmation.

- EC18 - Missing or empty confirm password
  - Input: Confirm new password
  - Type: Invalid
  - Class: Confirmation is omitted, empty, null, or only whitespace.
  - Reason: Confirmation is required to verify the new password entry.

- EC19 - Confirm password with incorrect data type
  - Input: Confirm new password
  - Type: Invalid
  - Class: Confirmation is not a string.
  - Reason: Confirmation must be compared exactly against the string new password.

- EC20 - Valid reset input combination
  - Input: Email address, OTP, new password, confirm new password
  - Type: Valid
  - Class: Registered email, OTP valid for that email, strong new password, and matching confirmation.
  - Reason: Password reset succeeds only when all input domains and dependencies are valid together.

- EC21 - Invalid reset input combination
  - Input: Email address, OTP, new password, confirm new password
  - Type: Invalid
  - Class: One or more required inputs are invalid, or dependencies between inputs fail.
  - Reason: Any invalid field or failed dependency must prevent password reset.

## 4. Domain Testing Methodology

1. Study the requirements.

FR-03 was reviewed as a two-step process: requesting an OTP and resetting the password. The API examples were also reviewed to identify backend inputs. The UI requirement for confirm new password was retained because the system requirement explicitly includes it.

2. Identify all inputs.

The inputs were identified as email address, OTP/reset token, new password, and confirm new password. Email appears in both the OTP request and reset validation, either explicitly in the API or implicitly through stored flow state.

3. Determine valid domains.

For each input, the valid domain was defined from the requirement: registered valid email, system-generated OTP tied to the same email, strong new password, and matching confirmation. Dependencies were included because FR-03 defines cross-field behavior, especially OTP-to-email ownership and password confirmation.

4. Determine invalid domains.

Invalid domains were identified by negating each requirement and by considering required-field and type-safety failures. Examples include malformed email, unregistered email, invalid OTP format, OTP for another email, weak password, and mismatched confirmation.

5. Partition inputs into equivalence classes.

Each input domain was separated into valid and invalid equivalence classes. Classes were split when different rules could fail independently, such as email format versus email registration, or OTP format versus OTP ownership.

6. Verify completeness of the identified domains.

The final review checked that all FR-03 inputs and dependencies are represented. The analysis covers the happy path, missing/empty values, wrong data types, invalid formats, invalid dependencies, weak password rules, and confirmation mismatch. No boundary values or test cases were included, in accordance with the skill instructions.
